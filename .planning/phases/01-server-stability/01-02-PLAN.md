# Plan: 01-02 - Error Handling and Health Checks

## Goal
Improve error granularity in `server.py` and implement comprehensive health checks for core components (especially Neo4j) to ensure a robust startup sequence.

## Proposed Changes

### `src/synapse_mcp/server.py`
- Define a custom exception hierarchy for Project Synapse (or at least handle specific exceptions).
- Implement a `check_health()` method in `SynapseServer` that verifies:
    - Neo4j connectivity and schema readiness.
    - Wiki vault accessibility.
- Refactor tool-level `try/except` blocks:
    - Move from generic `Exception` to specific handling where possible.
    - Standardize error responses to include component context.
    - Use `traceback.format_exc()` only in logs, not returned to the user (unless in a debug mode).
- Enhance the `lifespan_context` to fail fast if health checks fail.

### `src/synapse_mcp/core/knowledge_graph.py`
- Add a `check_health()` method that runs a simple query (e.g., `RETURN 1`).

### `src/synapse_mcp/wiki/wiki_adapter.py`
- Add a `check_health()` method that verifies the vault directory exists and is writable.

## Verification Plan

### Automated Tests
- Run `pytest` to ensure no regressions.
- Add a new test file `tests/test_health.py` to verify health check logic.

### Manual Verification
- Start the server with Neo4j down (if possible) to verify it fails with an informative message.
- Start the server with an invalid wiki path to verify the health check catches it.

## Must-Haves
- [ ] Neo4j connectivity is verified at startup.
- [ ] Tool errors provide context (e.g., "Neo4j Error: ..." vs "Error: ...").
- [ ] Startup fails gracefully if critical components are missing.
